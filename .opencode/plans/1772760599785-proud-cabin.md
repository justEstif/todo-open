# Plan: Marketable Web UX Refresh (No Simple.css, Go-served, Existing APIs)

## Goal
Replace `simple.css` with a custom design system and ship a marketable, polished "minimal productivity" experience while keeping Go serving HTML/CSS/JS and **without backend API changes** in this phase.

## Confirmed Scope
- Visual direction: **Minimal productivity**
- API scope: **No API changes** (use existing endpoints only)
- Keep architecture boundaries intact (`internal/api` transport, `internal/core` domain, `internal/store` persistence)

## Current Baseline (from code review)
- Web UI is static + client JS:
  - `internal/api/web/static/index.html`
  - `internal/api/web/static/app.js`
  - `internal/api/web/static/simple.css` (current, to be removed/replaced)
- Go serves embedded static files:
  - `internal/api/web/embed.go`
  - `internal/api/router.go`
- UI currently uses:
  - `POST /v1/tasks`, `GET /v1/tasks`, `PATCH /v1/tasks/{id}` (title-only), `DELETE /v1/tasks/{id}`
- Existing but underused route:
  - `GET /v1/adapters` (can support trust/health UX)

## Implementation Plan

### 1) UX + Structure Redesign (HTML shell)
Update `internal/api/web/static/index.html` to introduce a clearer, product-like flow:
- Header with concise value proposition + lightweight operational state area
- Primary quick-add input with stronger visual priority
- Task list area with empty/loading/error states
- Secondary edit/details panel with clearer selected-task context
- Mobile-first section ordering and spacing for usability

### 2) Replace Simple.css entirely with custom styles
Create a new first-party stylesheet (for example `internal/api/web/static/app.css`) and stop loading `simple.css`:
- Define design tokens (color, spacing, radius, type scale)
- Build a restrained, professional "minimal productivity" visual language
- Add responsive layout rules for desktop + mobile
- Add explicit focus-visible styles and improved hover/active contrast
- Keep motion subtle and respect reduced-motion preferences
- Remove dependency on generic framework styles so the brand/look is fully owned

### 3) Improve interaction quality using existing APIs (JS)
Refactor `internal/api/web/static/app.js` to feel app-like without backend changes:
- Reduce jarring full-state transitions (optimistic local updates where safe)
- Add local sorting/filtering/search over fetched tasks
- Improve selection/edit flow (clear active item, inline status messages)
- Improve error handling by mapping API errors to actionable messages
- Add small reliability indicators powered by `GET /healthz` and/or `GET /v1/adapters`

### 4) Web Interface Guidelines compliance pass
Apply rules from fetched guidelines to all updated web files (`index.html`, CSS, JS):
- Accessibility: labels, keyboard behavior, semantic controls, aria-live for async status
- Focus states: visible and consistent `:focus-visible`
- Forms: `name`, `autocomplete`, proper input semantics
- Content/copy: ellipsis character, actionable error text, concise label clarity
- Interaction safety: confirm destructive delete action (or undo pattern)

### 5) Keep server wiring stable (only adjust if assets change)
If stylesheet asset structure changes, minimally update:
- `internal/api/web/embed.go`
- `internal/api/router.go`

### 6) Update tests to match new surface behavior
Adjust/add tests for updated static surface and expectations:
- `internal/api/web_integration_test.go` (asset references, key UI markers)
- Keep API integration tests intact unless frontend contract assumptions need test alignment:
  - `internal/api/tasks_integration_test.go`

## Verification Plan

### Automated
- `go test ./internal/api`
- `go test ./...`
- `go build ./...`

### Manual end-to-end
1. Run server: `go run ./cmd/todoopen-server`
2. Open `/` and validate:
   - clean layout on desktop and mobile widths
   - create/edit/delete flows remain functional
   - loading/empty/error states are clear
   - keyboard + focus behavior is visible and usable
   - destructive action has confirmation/guard
3. Validate API-backed status elements (`/healthz`, optional `/v1/adapters`) render without breaking task flows.

## Risks and Mitigations
- Risk: visual overhaul introduces regressions in CRUD flow
  - Mitigation: keep API calls and payloads unchanged; preserve integration coverage
- Risk: accessibility regressions while redesigning rapidly
  - Mitigation: do explicit guidelines pass before finishing
- Risk: CSS bloat or inconsistency
  - Mitigation: tokenized styles and component-level class structure
